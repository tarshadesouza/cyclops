## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] tsup DTS failed with composite:true inherited from tsconfig.base.json**

- **Found during:** Task 1 (first build attempt)
- **Issue:** tsup DTS invoked tsc with `--incremental` as a CLI flag (TS5074) and couldn't resolve imported files in composite project context (TS6307)
- **Fix:** Applied plan fallback — set `composite: false` and `incremental: false` in packages/core/tsconfig.json
- **Files modified:** packages/core/tsconfig.json

**2. [Rule 3 - Blocking] composite:false in core broke downstream tsc project references (TS6306, TS6310)**

- **Found during:** Task 1 (pnpm build after composite:false fix)
- **Issue:** packages/detectors, packages/ai, apps/api, apps/worker all referenced packages/core in their tsconfig references array; tsc requires `composite:true` and `noEmit:false` on referenced projects
- **Fix:** Removed `@cyclops/core` from tsconfig references in all four downstream packages. Consumers now resolve types via package.json `types` field pointing to dist/index.d.ts. Build order preserved by turbo's `^build` dependency.
- **Files modified:** packages/detectors/tsconfig.json, packages/ai/tsconfig.json, apps/api/tsconfig.json, apps/worker/tsconfig.json

## Decisions Made

| Decision | Rationale |
|----------|-----------|

| Remove core from downstream tsconfig references | TS6306/TS6310 — composite:false makes core ineligible as a tsc project reference target |
| Types resolved via dist/index.d.ts | standard npm package type resolution; turbo ensures dist exists before consumers build |
